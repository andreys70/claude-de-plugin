# JIRA Comment Format Templates

Used by the `jira-updater` agent throughout the 13-phase AuditPath onboarding flow.

**Key principle:**
- Ticket **body** = stakeholder-facing (Scope, Requirements, Acceptance Criteria)
- **Comments** = AuditPath engineering progress, checkpoint summaries, and final results

---

## Progress Comment (posted at the start of each phase)

Posted by `jira-updater` at the start of every phase. No engineer approval needed.

```
AuditPath | Phase {N}/13: {phase_name}
Started: {timestamp PST}

Build spec:
  Table:       {table_name}
  Domain:      {domain}
  Write mode:  {write_mode or "TBC at CP1"}
  Branch:      {branch}
  DM folder:   configs/finance_mm_dm/{domain}/
  SOX folder:  configs/finance_mm_sox/{domain}/
  Plugin:      AuditPath v0.1.0
```

Phase names:
```
1:  JIRA Intake
2:  Source Analysis
3:  DM Build
4:  DM Job Run
5:  Unit Test
6:  DQ Build
7:  DQ Job Run
8:  SOX Validation
9:  PR + Close-out
```

*(Checkpoints and manual pauses are not phases — they are gates between phases. They get their own Checkpoint Comment format below.)*

---

## Checkpoint Comment (posted at each of the 4 engineer checkpoints)

Posted by `jira-updater` when the orchestrator reaches a checkpoint. No approval needed — it is a record.

```
AuditPath | Checkpoint {1|2|3|4}: {checkpoint_name} — awaiting engineer decision

{checkpoint_summary}
```

Checkpoint names:
- **CP1:** Source Analysis Go / No-Go
- **CP2:** DM Build + Unit Test Review
- **CP3:** DQ Conf Review
- **CP4:** Validation Results Review

---

## Final Results Comment (posted at CP4 after engineer approval)

Always shown to engineer before posting.

```markdown
## AuditPath SOX DQ Validation Results — {table_name}
**Period:** {start_date} to {end_date} | **Run date:** {dm_run_date} | **Domain:** {domain}

### Completeness
| Metric | Value |
|--------|-------|
| Source count | {src_count:,} |
| Target count | {tgt_count:,} |
| Delta | {delta:,} |
| Status | ✅ Full Match / ⚠️ Expected Tolerance / ❌ Mismatch |

### Accuracy (sample: {sample_n} records)
| Metric | Value |
|--------|-------|
| Matched | {matched_count:,} / {total_records:,} |
| Mismatched columns | {col_list or "None"} |
| Match % | {match_pct:.1f}% |
| Status | ✅ 100% / ❌ {match_pct:.1f}% |

### Artifacts
| Artifact | Details |
|----------|---------|
| DM conf | `configs/finance_mm_dm/{domain}/{table_name}.conf` |
| DQ conf | `configs/finance_mm_sox/{domain}/dq_{table_name}.conf` |
| Branch | `{branch}` |
| BPP DM pipeline | `{dm_pipeline_name}` — run ID: {dm_run_id} |
| BPP DQ pipeline | `{dq_pipeline_name}` — run ID: {dq_run_id} |
| PR | {pr_url} |
| Auto-fix attempts | DM: {dm_fix_attempts}/2 | DQ: {dq_fix_attempts}/2 |

{tolerance_note_if_applicable}

*Generated by AuditPath v0.1.0*
```

---

## Tolerance Note (append to Final Results Comment when delta > 0 but explained)

```
> ⚠️ Tolerance Note: {delta:,} source records ({delta_pct:.2f}%) arrived after DM pipeline
> run date ({dm_run_date}). These are expected late-arriving records.
> SOX compliance verdict: **PASS with expected tolerance**.
```

---

## Auto-Fix Loop Notification (posted during unit-test or DQ validation retry)

```
AuditPath | Auto-fix attempt {attempt}/2 — {DM unit test | DQ validation}
Issue:  {error_summary}
Action: {fix_description — which CTE / column / cast was changed}
Re-running {DM | DQ} pipeline...
```

---

## Escalation Notice (posted after 2 failed retries — no approval needed)

```
AuditPath | ⚠️ Escalation Required
Auto-fix loop exhausted (2/2 attempts failed).
Table: {table_name} | Domain: {domain}

Diagnostic:
{full_diagnostic from validator}

Manual engineer investigation required. AuditPath has stopped.
```
